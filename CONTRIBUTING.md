# Contributing to AIR

Thank you for your interest in AIR.

AIR is still in the early stage, so contributions in architecture, implementation, docs, testing, and community building are all useful.

## Ways to Contribute

- Open an issue for bugs, gaps, or design questions
- Propose architecture improvements
- Implement MVP modules such as `session`, `vm`, `store`, or `guest agent`
- Improve docs, diagrams, examples, and onboarding content
- Help shape the public roadmap

## Before You Start

- Read [README.md](README.md) for the project direction
- Read [ROADMAP.md](ROADMAP.md) for current priorities
- Read the docs in [`docs/`](docs) for product and architecture context

## Contribution Rules

- Keep changes focused and reviewable
- Prefer small pull requests over large unrelated bundles
- Match the repository structure and naming conventions in [AGENTS.md](AGENTS.md)
- Update documentation when changing architecture or public behavior
- Add tests when code paths become executable

## Recommended Workflow

1. Open an issue or start a discussion for non-trivial changes
2. Fork the repo and create a branch from `main`
3. Make a focused change
4. Update docs if needed
5. Submit a pull request with clear context

## Pull Request Expectations

Include:

- what changed
- why it changed
- scope and tradeoffs
- test evidence if code is involved
- follow-up work if the change is partial

## Good First Contribution Areas

- CLI skeleton
- session state store
- VM lifecycle wrapper
- guest agent protocol draft
- docs cleanup and examples
- roadmap issue breakdown

## Community Direction

We want AIR to become an open technical project around:

- AI sandboxing
- VM-based isolation
- agent runtime infrastructure
- secure code execution

If you want to help shape the project direction early, issues and design discussions are especially valuable.
