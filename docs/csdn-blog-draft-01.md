# 我为什么要开源 AIR：为 AI Agent 提供默认安全的执行环境

## 一、问题不是“运行代码”，而是“运行不可信代码”

这几年，AI 工具的能力变化非常快。它们已经不只是生成代码，而是开始直接执行代码、修改文件、跑测试、返回结果。

这会带来一个基础设施层面的新问题：

> 当代码是 AI 刚刚生成出来的，而且默认不可信时，我们应该把它放在哪里运行？

很多系统的第一反应是容器。但容器本质上仍然是共享宿主机内核，它更适合“运行自己的代码”，而不是“运行可能带有风险的代码”。

所以对 AI Agent 场景来说，我们真正需要的不是一个普通部署环境，而是一个更强的执行边界。

## 二、AIR 想解决什么

AIR，全称 Agent Isolation Runtime。

它的目标很直接：

> 为 AI Agent 提供一个默认安全、可销毁、可复现的执行环境。

在 AIR 的设计里，有几个核心原则：

- 默认隔离
- 默认无网络
- 默认可销毁
- 默认限制资源

这意味着 AI 生成的代码不会直接触碰宿主环境，而是在一个轻量 VM 中运行。

## 三、为什么是 VM，而不是容器

原因并不复杂。

容器的优点是快、轻、生态成熟，但它共享宿主机内核。对普通业务应用来说，这通常够用；但对 AI Agent 来说，这个假设不再稳妥。

因为你无法提前完全信任将要执行的代码。

VM 的价值在于：

- 每个任务有独立内核
- 执行边界更清晰
- 执行完成后可以直接销毁
- 后续可以通过 snapshot 提升恢复速度

这正好适合 AI Agent 的执行模型。

## 四、AIR 的产品方向

AIR 当前规划两种能力。

### 1. 一次性执行

```bash
air run hello.py
```

背后的流程是：

```text
创建 VM -> 注入任务 -> 执行 -> 返回结果 -> 销毁 VM
```

### 2. 有状态 Session

```bash
air session create
air session exec <id> "echo hello > a.txt"
air session exec <id> "cat a.txt"
air session delete <id>
```

这类能力适合持续工作的 Agent，因为它需要保留文件状态、上下文和中间结果。

## 五、技术路线怎么走

AIR 不会一开始就追求“大而全”，而是按阶段推进。

### 第一阶段：做出最小可跑版本

- session create
- session exec
- session delete
- 单机运行
- 本地状态存储
- Host/Guest 先用文件通信

### 第二阶段：补齐工程能力

- Guest Agent
- virtio-serial 或 vsock
- HTTP API
- overlay 文件系统
- timeout、GC、日志

### 第三阶段：做性能与平台能力

- snapshot / restore
- 预热 VM 池
- 流式输出
- 白名单网络
- 指标监控

## 六、为什么选择开源

AIR 不是一个只靠单点实现就能完成的项目。

它横跨：

- 虚拟化
- Guest/Host 通信
- 安全隔离
- 调度与生命周期
- AI Agent 基础设施

这类项目天然适合开源，因为它需要更多开发者一起打磨方向、架构和实现细节。

我希望 AIR 能逐步吸引这些方向的开发者加入：

- 做 Hypervisor 和 VM 管理的人
- 做 Go 控制面和 API 的人
- 做 Guest Agent 的人
- 关注 AI 安全执行的人
- 想一起把 Agent Runtime 做成基础设施的人

接下来的重点，是把这些设计变成一个真正可运行的 MVP。
